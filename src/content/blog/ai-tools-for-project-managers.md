## Execution Utility: 30-Minute Setup Checklist (Pass/Fail)

Use this checklist to go from “we have a tool” to “we have a repeatable workflow.” Each item is designed to be testable in one working session.

**How to use it:** pick one team + one active project, then run the checklist end-to-end. Any **Fail** becomes your fix list.

### 1) Define the single source of truth

- **Pass** if one system is clearly the source of truth for task status (e.g., Motion, Trello, Jira, Asana) and everyone agrees.
- **Fail** if tasks live in multiple places (Slack threads, docs, emails) with no “final” board.

### 2) Create a standard workflow (statuses + owners)

- **Pass** if every task has an owner and uses the same 4–6 statuses (example: Backlog → Ready → Doing → Blocked → Review → Done).
- **Fail** if the team uses inconsistent statuses or tasks don’t have owners.

### 3) Convert one meeting into action items automatically

- **Pass** if you can run a meeting, then within 10 minutes produce:
  - at least **5** action items,
  - each assigned to an owner,
  - each with a due date,
  - and posted into the source-of-truth system.
- **Fail** if actions stay in notes or require manual copy/paste for most items.

### 4) Set up “no-surprises” reporting

- **Pass** if you can generate a weekly status update in under **15 minutes** that includes:
  - progress since last update,
  - current risks/blockers,
  - next milestones (with dates),
  - and clear asks/decisions.
- **Fail** if reporting depends on chasing people in DMs.

### 5) Add lightweight risk signals

- **Pass** if the system can surface at least **2** of these automatically: overdue tasks, blocked work, scope changes, workload conflicts, slipping milestones.
- **Fail** if you only find risks in the final week.

### 6) Validate discoverability and on-page SEO (optional, 5 minutes)

If this guide is part of a publishing workflow (internal wiki or public blog), run a quick on-page check so the page is easy to find.

- **Pass** if the page has a clear title/H1 match, specific description, and at least 2 relevant internal links.
- **Fail** if the page is vague (“Project update”) or has no internal links.

Run our internal tool: [SEO Audit Tool](/tools/seo-audit).

### 7) Final acceptance test (the “Friday test”)

- **Pass** if on Friday, a stakeholder can answer these in under 2 minutes from the system:
  - What shipped this week?
  - What is blocked and why?
  - What are the top 3 priorities next week?
- **Fail** if someone must compile the answer manually.

**If you failed 3+ items:** don’t buy more tools yet—simplify the workflow first, then re-run the checklist.
