## Why this happens (fast diagnosis)

If you searched for **“pip install keeps installing globally”** or **“pip installs to global site-packages”**, you’re usually in one of these situations:

- You created a virtual environment but didn’t activate it.
- Your editor/terminal is using a different Python interpreter than you think.
- `pip` is mapped to a different Python (common on macOS/Windows).
- Your environment variables or `pip` config are forcing user/global installs.

This guide gives you a **repeatable checklist** to ensure installs land in the right place.

## 60-second checklist (copy/paste)

Run these inside your project folder:

```bash
python -c "import sys; print(sys.executable)"
python -m pip --version
pip --version
python -c "import site; print(site.getsitepackages()); print(site.getusersitepackages())"
```

You want `sys.executable` to point to your project’s `.venv` (or `venv`) path.

## Fix 1: Activate the virtual environment correctly

### macOS/Linux

```bash
python -m venv .venv
source .venv/bin/activate
python -m pip install -U pip
python -m pip install requests
```

### Windows (PowerShell)

```powershell
py -m venv .venv
.\.venv\Scripts\Activate.ps1
py -m pip install -U pip
py -m pip install requests
```

If activation works, `which python` (macOS/Linux) or `where python` (Windows) should show the venv path.

## Fix 2: Always use `python -m pip` (avoids “wrong pip”)

Using `pip` directly can point at a different interpreter than `python`.

Prefer:

```bash
python -m pip install <package>
```

If you have both Python 3 and Python 2 remnants, use:

```bash
python3 -m pip install <package>
```

## Fix 3: VS Code is using the wrong interpreter

Symptoms:

- Terminal installs work, but imports fail in the editor.
- VS Code shows a different Python version than your terminal.

Fix:

1. `Cmd/Ctrl+Shift+P` → **Python: Select Interpreter**
2. Choose the interpreter inside your project: `.venv/bin/python` (or `\.venv\Scripts\python.exe`)
3. Reload window

## Fix 4: `pip config` or env vars are forcing user/global installs

Check config:

```bash
pip config list
```

Look for:

- `global.user = true`
- custom `target` directories

Also check env vars:

```bash
python -c "import os; print(os.environ.get('PIP_USER'), os.environ.get('PIP_TARGET'))"
```

If you see `PIP_USER=1`, your installs may bypass the venv.

## Fix 5: You’re in a PEP 668 “externally managed environment”

On some systems (notably newer Debian/Ubuntu), you’ll get errors or unexpected behavior with system Python.

Best practice:

- Use a project venv for apps.
- Use `pipx` for global CLI tools.

## Better defaults (2026 workflow)

### Option A: `uv` for fast, reproducible installs

If you’re already using `uv`, prefer:

```bash
uv venv
uv pip install <package>
```

### Option B: `pipx` for global CLI tools

Examples:

```bash
pipx install black
pipx install ruff
```

## Quick self-test: prove the package is installed in the venv

```bash
python -c "import pkgutil; print('requests' in [m.name for m in pkgutil.iter_modules()])"
```

If it prints `True`, your interpreter can see the package.
